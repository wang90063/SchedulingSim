第一版业务模型要抽象到哪一层？

看成“平台内部的核心对象长什么样”，不是界面层

A. 通用实验模型
User / Task / Edge Node / Result
用户产生任务 → 调度器分配到边缘节点 → 输出时延/吞吐/成功率

偏平台化

核心对象很通用，后面接不同业务和算法都方便，但第一版对你当前调度问题的贴合度会弱一些。

B. 贴近当前调度研究
UE / LC / PDB / candidate_ue / Edge
UE 与业务流进入候选队列 → 算法决定顺序 → 统计尾时延、抖动、公平性

偏研究验证

第一版最贴近你现在的问题，能更快验证算法思路，但未来切到别的业务场景时需要再抽象一次。

C. 双层模型
Core Model + Cellular Adapter
底层统一实验对象，上层用蜂窝调度适配器表达 UE / LC / PDB 等细节

平台与研究兼顾

平台骨架先稳定，当前调度问题也能表达。复杂度比 A/B 高一点，但更符合“模块化、低耦合”的目标。

我目前更推荐 C

  • 实验管理层只依赖通用实验模型
  • 环境仿真层只依赖标准接口
  • 你的算法和基线算法都挂在适配层里

代价

  • 第一版设计要多做一层接口
  • 需要先定义清楚核心对象边界
