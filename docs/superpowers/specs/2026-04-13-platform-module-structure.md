# 当前平台模块结构说明

## 总体分层

当前代码已经形成比较清晰的三层：

1. **实验配置 / 场景层**
2. **无线环境 / 调度仿真层**
3. **指标报表层**

你的算法以后主要落在“排序、回插、分配”几个模块里，无线环境和报表可以相对独立演进，耦合不算重，不需要推倒重写。

## 模块说明

### `src/scheduling_sim/config.py`

职责：

- 解析 JSON 配置
- 定义实验配置数据结构
- 兼容 `legacy` 和 `UMa` 两类无线环境 schema

后续适合扩什么：

- 新的业务参数
- 新的无线环境参数
- 新的算法开关

不建议放什么：

- 运行时逻辑
- 调度算法实现

### `src/scheduling_sim/models.py`

职责：

- 定义核心数据对象
- 统一用户、LC、包、无线快照、调度 grant 的结构

这是全平台的“公共语言层”。如果未来要加新算法，优先加字段，不要把算法逻辑塞进这里。

### `src/scheduling_sim/scenario.py`

职责：

- 根据配置生成用户
- 固定 `1 UE = 1 LC`
- 给中心/边缘用户灌入业务模型和初始无线侧属性

你以后如果想把“用户类型”再细化，比如中心、次边缘、极边缘，主要改这里和配置层。

### `src/scheduling_sim/wireless_env.py`

职责：

- 按 slot 刷新无线快照
- 目前实现的是 `UMa` 大尺度上行环境
- 输出 `snr_db / mcs_index / bits_per_prb / per_u_slot_prb_cap`

这里是“无线细节隔离层”。排序、回插、PRB 分配不需要知道路径损耗公式，只吃 `CurrentRadioState`。

### `src/scheduling_sim/queue.py`

职责：

- 维护全局活动队列
- 提供队头 `K` 个候选用户的窗口视图
- 支持尾插和指定位置插入

这是回插算法的直接操作对象。

### `src/scheduling_sim/ranking.py`

职责：

- 负责候选集内排序
- 当前实现是 `EPF`

你后续如果加新排序算法，最自然的做法就是在这里并列加实现，然后通过配置切换。

### `src/scheduling_sim/planning.py`

职责：

- 把 `D/S` 两个决策 slot 的资源预算映射到后面三个 `U` slot
- 根据排序结果顺序分配 `PRB`

当前模型已经符合我们对齐的简化语义：

- `D` 用前半段 `PRB`
- `S` 用后半段 `PRB`
- 后面 `U` 只执行，不再做新决策

### `src/scheduling_sim/reinsert.py`

职责：

- 定义回插策略
- 目前有 `tail_append` 和 `constrained_insert`

这是你算法的核心落点之一。以后如果做更复杂的“预测进入候选集时间 / 估算消费速度 / 最远可满足 PDB 插入点”，最好继续放在这里，不要污染 `simulator.py`。

### `src/scheduling_sim/simulator.py`

职责：

- 编排整个 `DSUUU` 仿真循环
- 调用队列、排序、规划、回插、无线环境、包消费、HOL 刷新
- 在物理 `U-slot` 上合并 `D/S` grant，并执行边缘 `PRB cap`

这个文件更像总控，不应该承载算法细节。现在的结构基本合理，后面只需要逐步把算法实例化方式再抽象一点。

### `src/scheduling_sim/metrics.py`

职责：

- 收集完成包、发送 bit、按用户统计等原始指标
- 汇总成中心 `GBR` 和边缘 delay / `HOL` 业务 KPI

以后如果你要做更多论文口径指标，比如 Jain 指数、边缘分位数时延、业务满足率组合分数，也建议先放这里。

### `src/scheduling_sim/reporting.py`

职责：

- 把 summary 写成 JSON

保持简单就好，不建议把分析逻辑塞进这里。

### `src/scheduling_sim/cli.py`

职责：

- 提供命令行入口
- 跑一次实验，输出一份 report

以后如果你要做批量 sweep、多算法对比、多配置实验矩阵，可以从这里再往外扩。

## 当前扩展点建议

如果按你后面的研究路线，我建议优先把扩展点固定成这三个接口面：

1. **排序算法扩展点**：`ranking.py`
2. **回插算法扩展点**：`reinsert.py`
3. **业务模型扩展点**：`scenario.py` + `config.py`

这样改一个地方，主流程基本不用动，符合你前面说的“模块化、低耦合、在一个地方改完就能生效”。

## 我对当前架构的判断

结论很明确：

- **不用重开发**
- **也不用现在大重构**
- 当前架构已经够支撑第一版实验平台

真正值得继续做的，不是“重写框架”，而是：

1. 把 `ranking/reinsert/planning` 的实例化方式再抽成更统一的策略接口
2. 把实验批跑和对比分析工具补上
3. 把业务 KPI 口径继续稳定下来

这三件事的收益会明显高于重开项目。
