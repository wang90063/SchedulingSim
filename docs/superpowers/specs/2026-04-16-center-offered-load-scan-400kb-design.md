# 400KB Target Edge 场景中的中心业务负载扫描设计

## 背景

我们已经验证了当前 `400KB` 目标边缘大包实验里，一个重要干扰项来自中心业务过碎：

- 旧口径使用 `160 bit`、`period_slots = 1`
- 这会让中心用户在每个 `U-slot` 都注入极碎小包
- 在 PRB 分配按包向上取整、但实际发送 bit 又受包剩余量限制的机制下，会放大资源切片浪费与排队碎片化

因此，之前观察到的“中心吞吐变化很大，但边缘大包完成时间没有同步变长”并不代表系统凭空多出了资源，而是说明极碎小包场景里存在明显的 packing waste。

基于这个结论，后续主口径不再继续使用 `160 bit / every 1 slot`，而改为：

- `960 bit / every 6 slots`

这样仍然保持中心背景业务较轻，但能显著减少碎片化带来的解释偏差。

在这个新默认口径下，本轮还要继续回答另一个更直接的问题：

> 当中心用户传输的总 bit 数变多时，目标边缘大包的完成时延会不会明显变长？

也就是说，这一轮不再验证“碎不碎”，而是验证“在较合理颗粒度下，中心业务 offered load 变重是否真的会挤压边缘大包时延”。

## 目标

本轮新增实验需要回答以下问题：

1. 在默认中心业务改为 `960 bit / every 6 slots` 后，原有 `400KB` 章节的结论是否保持稳定。
2. 当中心用户数固定、发送周期固定为 `6 slots`、单包 bit 数逐步增大时，目标边缘大包的 `Completion Delay` 是否显著上升。
3. 两种策略在中心 offered load 提升后，边缘收益与中心吞吐代价是否同步变化。
4. 在较低碎片化环境下，边缘时延变化是否更能反映“真实负载竞争”而不是“包太碎带来的资源浪费”。

## 范围

本轮只覆盖以下范围：

- 目标边缘包保持为原报告中的 `400KB` 章节
- 沿用原报告中的两种策略对比：
  - `tail_append`
  - `business_aware_constrained_insert`
- 保留现有主指标：
  - `Completion Delay`
  - `Queue Wait`
  - `Service Time`
  - `Center Avg Rate`
- 在原有 `400KB` 章节基础上追加一节新的“中心业务负载扫描”
- 将主配置中的默认中心业务口径调整为 `960 bit / every 6 slots`

本轮明确不做：

- 不新增新的吞吐定义或统计窗口
- 不改停止条件，仍然按当前目标边缘包完成时刻截断统计
- 不把新负载扫描扩展到 `800KB ~ 2000KB` 所有章节
- 不引入新的调度算法或新的无线参数扫描

## 核心设计

### 1. 将默认中心业务口径改为 `960 bit / every 6 slots`

原报告默认环境中的中心业务从：

- `160 bit / every 1 slot`

改为：

- `960 bit / every 6 slots`

这样做的目的不是提升系统负载，而是去掉已经确认存在的碎片化偏差，使原报告其余章节都建立在更合理的背景业务颗粒度之上。

### 2. 新增“中心业务负载”扫描维度

在 `400KB` 章节中新增一个只改变中心单包大小的扫描维度，并保持：

- `center_user_count = 63` 固定
- `period_slots = 6` 固定

扫描 `packet_bits`，从而让中心 offered load 随 `packet_bits / period_slots` 单调增加。

建议档位：

- `960 bit / every 6 slots`
- `1440 bit / every 6 slots`
- `1920 bit / every 6 slots`
- `2880 bit / every 6 slots`

这组档位对应的含义清晰：

- `960` 作为新的默认基线
- `1440` 表示在固定 6-slot 周期下小幅增载
- `1920` 表示约 `2x` 基线 offered load
- `2880` 表示约 `3x` 基线 offered load

如果后续发现 `2880` 仍然不足以拉开边缘时延差距，再单独追加更高档位；本轮先保持报告可读性。

### 3. 结果解释方式

新增扫描后，解释顺序应改成：

- 先看 `Completion Delay` 是否随中心业务 load 上升而单调变长；
- 再看时延增加主要来自 `Queue Wait` 还是 `Service Time`；
- 最后看两种策略在更高中心负载下，中心平均吞吐与边缘收益之间是否出现新的 trade-off。

如果边缘完成时延随着 `packet_bits` 增长明显上升，就说明在新的较低碎片化环境下，中心总 bit 数确实会挤压边缘服务进度；
如果变化仍然不明显，则说明当前 `400KB` 场景对额外中心负载还不敏感，或者边缘用户由于权重与资源上限机制，仍能较快完成。

## 报告设计

在 `400KB` 章节内新增一节：

- `### 中心业务负载扫描`

每一行对应一个固定 `period_slots = 6` 的中心业务档位，展示：

- `Baseline Completion`
- `Ours Completion`
- `Baseline Queue Wait`
- `Ours Queue Wait`
- `Baseline Service`
- `Ours Service`
- `Baseline PDB`
- `Ours PDB`
- `Baseline Center Avg`
- `Ours Center Avg`
- `结论`

章节后新增一段趋势总结，重点解释：

- 中心 offered load 增加后，边缘大包完成时延是否同步拉长；
- 时延增长主要是排队等待变长，还是服务被打散；
- 两种策略在高负载下是否继续保持相对收益。

## 配置与代码变更方向

### 配置

需要调整 `configs/target_edge_packet_size_sensitivity_main_uncapped.json`：

1. 把默认中心业务改为：

```json
"center": { "count": 63, "period_slots": 6, "packet_bits": 960, "pdb_ms": 1000000000, "gbr_bps": 7000 }
```

2. 在 `sweep` 中新增一个只用于 `400KB` 章节的负载扫描字段，例如：

```json
"center_offered_load_packet_bits": [960, 1440, 1920, 2880]
```

这里不需要再扫 `period_slots`，因为这轮实验的目标就是在固定 `6 slots` 周期下，单独验证总 bit 数增加的影响。

### 脚本

需要扩展 `scripts/run_target_edge_packet_size_sensitivity_report.py`：

- 支持新的 `center_offered_load_packet_bits` 维度
- 在 `400KB` 章节中额外生成对应 rows
- 给新维度生成 `value label`，例如 `1920 bit / every 6 slots`
- 在 markdown 中新增“中心业务负载扫描”章节与趋势分析段落
- 同时把章节中的固定环境描述改成新的默认口径 `960 bit / every 6 slots`

### 测试

需要扩展 CLI smoke test，至少覆盖：

1. stdout 中出现 `center_offered_load` 维度的新 rows；
2. markdown 报告中出现“中心业务负载扫描”章节；
3. 章节中出现 `960 / 1440 / 1920 / 2880 bit / every 6 slots` 的标签；
4. 原有章节仍保留且脚本能完整跑通。

## 风险与取舍

### 风险 1：默认口径变化会影响原报告数值

这不是纯追加实验，而是把原报告默认中心业务背景从 `160/1` 切换到 `960/6`。因此：

- 原有 `PDB` 扫描与中心用户数扫描的绝对数值会变化；
- 但这正是本轮想要的结果，因为旧默认口径已经被确认会放大碎片化偏差。

### 风险 2：`Center Avg Rate` 仍不是最终吞吐定义

当前吞吐仍按目标边缘包完成时刻截断，因此它更适合作为“伴随指标”，不适合作为本轮唯一结论来源。
本轮主问题是“边缘时延是否随中心总 bit 数增加而变长”，因此结论优先围绕 `Completion Delay` 与 `Queue Wait` 展开。

### 风险 3：负载扫描档位过多会降低可读性

如果扫太多点，报告会迅速变宽、结论也会被稀释。因此本轮先采用 `4` 个档位，确保趋势可读。

## 验证

实现后至少验证以下几点：

1. 默认中心业务配置已经从 `160/1` 切换到 `960/6`；
2. 新增 `center_offered_load_packet_bits` 配置能被脚本正确解析；
3. 新维度只在 `400KB` 章节执行，不会扩散到所有包大小；
4. stdout、CSV、JSON、Markdown 都能看到新维度结果；
5. 原有 `PDB` 与中心用户数章节仍能正常生成；
6. 新报告文案明确区分“默认口径变更”和“400KB 额外负载扫描”这两件事。

## 预期结论

这轮实验的预期不是证明某个策略必然更优，而是回答一个更直接的问题：

- 在去掉明显碎片化偏差后，中心业务总 bit 数变多，是否真的会让边缘大包更慢完成；
- 如果会，增长主要体现在哪类等待时间上；
- 如果不会，那么当前 `400KB` 场景里的主要矛盾就不在“中心总 bit 数不够多”，而可能仍在调度窗口、资源分配粒度或边缘优先级结构本身。
