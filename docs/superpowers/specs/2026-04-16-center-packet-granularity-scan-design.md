# 400KB Target Edge 场景中的中心业务颗粒度扫描设计

## 背景

当前 `400KB` 目标边缘大包实验中，中心背景业务使用 `160 bit` 小包，且 `period_slots = 1`。这意味着中心用户在每个 `U-slot` 都会注入一个非常碎的小包。现有结果里，中心用户平均吞吐与目标边缘包完成时延之间存在较明显差异，但这里混合了两类机制：

1. 算法本身的排队与重插入影响；
2. 中心业务颗粒度过碎带来的候选窗口、轮转排队和资源切片放大效应。

如果直接把中心小包改成更大的值，例如 `1000 bit`，并保持 `period_slots = 1` 不变，那么中心 offered load 会同步大幅上升。这会把“业务颗粒度变化”和“系统负载变重”混在一起，无法回答当前真正关心的问题：

> 现在看到的吞吐/时延差异，有多少来自算法本身，又有多少来自 `160 bit` 小包过碎的环境设定？

因此，本轮设计先不改变边缘目标包，不改变主要算法，也不改变实验主口径，而是在 `400KB` 场景中额外加入一个“中心业务颗粒度扫描”维度，并尽量保持中心平均 offered load 近似不变。

## 目标

本轮新增实验需要回答以下问题：

1. 当中心业务保持近似相同平均 offered load，但从“极碎小包”变成“更整块的包”时，目标边缘包的完成时延、等待时延和服务轨迹会如何变化。
2. 当中心包不再那么碎时，当前报告里观察到的中心吞吐差异是否会缩小。
3. 当前现象是否主要由 `160 bit` 小包对队列轮转和候选窗口机制造成的放大效应驱动。

## 范围

本轮只覆盖以下范围：

- 目标边缘包保持为当前主场景的 `400KB`
- 只在现有主报告上增加一个新的扫描章节
- 保留当前两种策略对比：
  - `tail_append`
  - `business_aware_constrained_insert`
- 保留当前 `completion delay / queue wait / service time / center_avg_rate_bps` 等核心指标

本轮明确不做：

- 不扩展到更大目标边缘包尺寸
- 不把该维度加入所有其它实验矩阵
- 不修改停止条件或吞吐定义
- 不引入新的调度算法

## 核心设计

### 1. 新增“中心业务颗粒度”扫描维度

新增一个由 `(center_packet_bits, center_period_slots)` 组成的组合维度，用来表达：

- 单包大小变大
- 到达周期同步拉长
- 从而使中心平均 offered load 保持近似不变

本轮推荐的扫描档位如下：

- `160 bit, every 1 slot`
- `320 bit, every 2 slots`
- `480 bit, every 3 slots`
- `800 bit, every 5 slots`
- `960 bit, every 6 slots`

这些档位的共同特点是：

- 都能整除当前基线 offered load 口径
- 便于直接解释“颗粒度变化”
- 避免使用 `1000 bit` 这类不方便做等负载对应的数值

### 2. 保持其它环境参数不变

除中心业务颗粒度外，其它参数沿用当前 `400KB` 主场景配置：

- `resources.total_prb_per_u_slot = 237`
- `max_ue_per_slot = 16`
- 目标边缘大包维持当前大小
- 无线环境仍采用当前 UMa 设定
- 中心用户数、边缘 PRB cap、目标 PDB 采用当前主报告中的默认口径

这样做的目的是把影响尽量隔离到“中心业务颗粒度”本身，而不是重新引入新的干扰因子。

### 3. 结果解释方式

新增扫描维度后，主解释逻辑如下：

- 如果中心吞吐和目标时延差异明显收敛，说明之前的结果确实受到了超碎小包设定的放大；
- 如果差异仍然稳定存在，说明当前现象更多反映了算法与候选窗口机制本身，而不是 `160 bit` 小包的偶然产物；
- 如果目标边缘包虽然更早进入候选集，但在更大中心包背景下分到的资源并没有明显增加，则说明“进候选窗口”和“真正拿到足够资源”是两个不同层次的问题。

## 报告设计

主报告中新增一节，例如：

- `## 中心业务颗粒度扫描`

每一行对应一个 `(center_packet_bits, center_period_slots)` 组合，输出：

- `Baseline Completion`
- `Ours Completion`
- `Baseline Queue Wait`
- `Ours Queue Wait`
- `Baseline Service`
- `Ours Service`
- `Baseline Center Avg`
- `Ours Center Avg`
- 差异结论

章节后补一段趋势总结，重点回答：

- 小包越碎时，候选窗口轮转效应是否更明显
- 包更整块后，边缘大包是否更容易“看得见但拿不到”
- 当前吞吐差异是否缩小

## 配置与代码变更方向

### 配置

需要在 `400KB` 主实验配置中增加新的 sweep 字段，用于描述颗粒度档位。推荐采用显式成对结构，而不是两个独立数组，避免错误组合。

建议形态示意：

```json
"center_packet_granularity": [
  { "packet_bits": 160, "period_slots": 1 },
  { "packet_bits": 320, "period_slots": 2 },
  { "packet_bits": 480, "period_slots": 3 },
  { "packet_bits": 800, "period_slots": 5 },
  { "packet_bits": 960, "period_slots": 6 }
]
```

### 脚本

需要扩展现有 `run_target_edge_packet_size_sensitivity_report.py`，使其在 `400KB` 章节内额外执行该维度扫描，并将结果写入：

- JSON
- CSV
- Markdown report

因为这个实验只针对 `400KB` 主场景，所以实现上应当把颗粒度扫描限定在该包大小章节内，避免无意中把扫描矩阵扩到所有包大小。

### 指标

本轮继续使用当前已有指标，不新增新的吞吐定义。这样可以先回答“环境口径是否放大了结果”，而不是在同一轮里同时改实验结构和改业务指标。

如果这一轮结果仍然显示吞吐解释存在明显偏差，再单独设计第二轮：把“固定窗口内中心总吞吐 / 人均吞吐”作为新的主口径。

## 风险与取舍

### 风险 1：`center_avg_rate_bps` 仍然可能带有解释偏差

虽然本轮不改吞吐定义，但扫描目标是缩小环境设定造成的偏差，而不是彻底解决吞吐口径问题。因此报告里需要明确：

- 本轮是在现有吞吐定义下，观察环境颗粒度变化是否影响结论；
- 这不是新的最终吞吐评价体系。

### 风险 2：扫描矩阵膨胀

如果把颗粒度维度直接并入所有 `PDB`、`center_user_count` 和包大小章节，实验规模会迅速膨胀，报告也会失去可读性。因此本轮只把它限定在 `400KB` 主场景。

### 风险 3：用户误解为“负载变了”

尽管这里是近似等 offered load 设计，但不同颗粒度下，真实调度行为、完成包数量和服务碎片化仍可能不同。报告文案应避免把它写成“完全相同负载”，而要表述为“平均 offered load 近似保持一致”。

## 验证

实现后应至少验证以下几点：

1. 新增 sweep 配置能被正确解析；
2. 每个颗粒度档位下，中心 `packet_bits` 与 `period_slots` 都按预期生效；
3. 报告新增“中心业务颗粒度扫描”章节；
4. 章节中能正确输出每档的两种策略对比结果；
5. 原有 `400KB` 主报告其余章节不被破坏。

## 预期结论

本轮实验的预期不是证明某个算法一定更优，而是校正解释环境：

- 如果颗粒度扫描后结果收敛，就说明当前 `160 bit` 设定确实放大了差异；
- 如果结果不明显收敛，就说明当前结论具备更强稳健性；
- 无论哪种结果，都能帮助后续决定是否还需要继续修改吞吐主口径。
