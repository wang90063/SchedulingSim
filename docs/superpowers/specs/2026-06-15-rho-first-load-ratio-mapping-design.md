# Rho-First Load-Ratio Mapping Design

## Goal

Replace the current load-ratio scene definition with a stricter two-stage design:

1. define the scan directly in `rho` space
2. map each `rho` point back into runnable business parameters

This change is meant to remove the ambiguity in the current first-pass design, where the scan is described as a load-ratio sweep but is actually constructed by fixing several business parameters first and only then deriving `rho`.

## Problem With the Current Design

The current implementation fixes:

- `N_bg = 40`
- `bg_period_ms = 10`
- `N_pdb = 4`

and then sweeps:

- background packet size
- `PDB` packet size under a few fixed `pdb_ms` shapes

This does produce distinct `rho_bg` and `rho_pdb` points, but it mixes two separate concerns:

- the intended scan coordinate system
- one specific business-parameter realization of that coordinate system

That makes the experiment harder to interpret. A reader can no longer tell whether a result came from:

- the `rho` point itself
- the mapping policy used to reach that `rho`

The scan definition should therefore move up one level of abstraction.

## Chosen Direction

Use a **rho-first scan with explicit back-mapping**.

The scan is defined only by:

- target `rho_bg`
- target `rho_pdb`
- a small `PDB` shape axis

Business parameters are then generated by a documented mapping policy.

This keeps the intended scan coordinates clean while still producing concrete simulator inputs.

## Alternatives Considered

### 1. Keep the current business-first sweep

Reject.

It is easy to run, but it makes the scan definition depend on one implicit mapping policy. That is the exact confusion this change is meant to remove.

### 2. Use rho-only scenes with no back-mapping policy

Reject.

The simulator cannot run directly on abstract `rho` values. Concrete business parameters are still required.

### 3. Use rho-first scenes with one explicit back-mapping policy

Choose this.

It preserves analytical clarity while keeping the first pass small and runnable.

## Fixed Supply Side

The wireless side remains fixed:

- `1 ms` slot duration
- `DSUUU` TDD pattern
- `273 PRB` per `U-slot`
- background users drawn from `medium + good`
- `PDB` users drawn from `poor`
- no wireless parameter is a sweep axis in this pass

This remains a business-pressure scan, not a radio sweep.

## Scan Coordinates

### Primary Axes

The scene grid is defined by:

- `rho_bg_values`
- `rho_pdb_values`

with:

- `rho_bg = L_bg / C_bg`
- `rho_pdb = L_pdb / C_pdb`

### Derived Mix Metric

`prb_share_pdb` is still reported, but it is not configured independently:

- `prb_share_pdb = rho_pdb / (rho_bg + rho_pdb)`

It remains a derived descriptor of class pressure mix.

### Secondary Axis

`PDB` shape remains a small explicit secondary axis:

- `pdb_ms_values = [20, 50, 100, 300, 500]`

This preserves the earlier design intent: the same `rho_pdb` can be realized under different `PDB` timing shapes.

## First-Pass Rho Grid

The first pass keeps the already-approved ratio coordinates:

- `rho_bg_values = [0.388, 0.582, 0.775, 0.969]`
- `rho_pdb_values = [0.183, 0.366, 0.549]`
- `pdb_ms_values = [20, 50, 100, 300, 500]`

This still yields:

- `4 √ó 3 √ó 5 = 60` scene points

The difference is that these are now the **input definition**, not a description of one specific parameterization.

## Capacity Reference

The current first pass continues to use the same calibration references:

- `C_bg = 66.03 Mbps`
- `C_pdb = 8.74 Mbps`

These are used only for:

- converting target `rho` values into offered load targets
- labeling the resulting scene points

They are not substitutes for simulated throughput.

## Back-Mapping Policy

Each class needs a policy that turns target load into concrete business parameters.

### Background Mapping Policy

Use a **candidate-domain solver**, not a single fixed template.

The background mapper enumerates:

- `background_user_count ‚ąą {32, 40, 48}`
- `background_packet_kb ‚ąą {1.0, 1.5, 2.0}`

and solves for:

- `background_period_ms`

using:

- `L_bg = rho_bg √ó C_bg`
- `background_period_ms = background_user_count √ó background_packet_kb √ó 8 / L_bg`

This policy is chosen because:

- the scan is still defined in `rho` space
- business realization remains flexible
- the background packet model does not have to be artificially fixed to one template

For reference, if the mapper used the anchor pair:

- `background_user_count = 40`
- `background_packet_kb = 2.0`

then the target `rho_bg` points would approximately imply:

- `0.388 -> 25.0 ms`
- `0.582 -> 16.7 ms`
- `0.775 -> 12.5 ms`
- `0.969 -> 10.0 ms`

These values are examples only, not the scan definition.

### PDB Mapping Policy

Use:

- candidate `pdb_user_count ‚ąą {4, 8}`
- fixed `pdb_ms`
- solve for `pdb_packet_kb`

The mapping is:

- `L_pdb = rho_pdb √ó C_pdb`
- `pdb_packet_kb = L_pdb √ó pdb_ms / (pdb_user_count √ó 8)`

This keeps the earlier shape axis while defining it from `rho_pdb` instead of choosing packet sizes first.

The first pass expands the `PDB` shape axis to:

- `pdb_ms_values = [20, 50, 100, 300, 500]`

Using the anchor choice `pdb_user_count = 4`, the target `rho_pdb` points would approximately imply:

- `rho_pdb = 0.183`
  - `20 ms -> 1 KB`
  - `50 ms -> 2.5 KB`
  - `100 ms -> 5 KB`
  - `300 ms -> 15 KB`
  - `500 ms -> 25 KB`
- `rho_pdb = 0.366`
  - `20 ms -> 2 KB`
  - `50 ms -> 5 KB`
  - `100 ms -> 10 KB`
  - `300 ms -> 30 KB`
  - `500 ms -> 50 KB`
- `rho_pdb = 0.549`
  - `20 ms -> 3 KB`
  - `50 ms -> 7.5 KB`
  - `100 ms -> 15 KB`
  - `300 ms -> 45 KB`
  - `500 ms -> 75 KB`

These values are examples only, not the scan definition.

### Why Only One Mapping Policy Family In This Pass

The point of this change is to separate:

- scan definition
- mapping realization

not to sweep multiple mapping policies at once.

Multiple mapping policies may be useful later, but they would multiply the scene count and mix two questions:

- where in `rho` space is there gain
- how sensitive is that gain to business realization style

This pass should answer the first question only.

The candidate domains above do not create new scan axes. They only provide the solver with a bounded set of acceptable business realizations for each target `rho` point.

## Candidate Domains And Solver Rules

### Candidate Domains

The first pass uses these realization domains:

- `background_user_count ‚ąą {32, 40, 48}`
- `background_packet_kb ‚ąą {1.0, 1.5, 2.0}`
- `pdb_user_count ‚ąą {4, 8}`
- `pdb_ms ‚ąą {20, 50, 100, 300, 500}`

Only `pdb_ms` is part of the scene definition. The other three are candidate inputs available to the mapper.

### Validity Constraints

The mapper must reject candidate realizations unless all of the following hold:

- `background_period_ms ‚ąą [4, 30]`
- `pdb_packet_kb ‚ąą [0.5, 80.0]`

These bounds are not theoretical limits. They are practical guardrails to avoid obviously unrealistic or unhelpful realizations in this pass.

### Selection Rule

For each target scene point:

1. enumerate all candidate background realizations
2. enumerate all candidate `PDB` realizations for the requested `pdb_ms`
3. round each realizable parameter to simulator-compatible values
4. recompute `actual_rho_bg` and `actual_rho_pdb`
5. choose the valid pair with the smallest total ratio error

where total ratio error is:

- `abs(actual_rho_bg - target_rho_bg) + abs(actual_rho_pdb - target_rho_pdb)`

### Tie-Break Preference

If multiple realizations have the same error, prefer the one closest to these anchor values:

- `background_user_count = 40`
- `background_packet_kb = 2.0`
- `pdb_user_count = 4`

This keeps the first pass deterministic and preserves comparability without redefining the scan around a single hard-coded business template.

## Scene Record Requirements

Every generated scene must record both the intended coordinates and the realized business parameters.

### Required Target Fields

- `target_rho_bg`
- `target_rho_pdb`

### Required Actual Fields

- `actual_rho_bg`
- `actual_rho_pdb`
- `prb_share_pdb`

### Required Mapping Metadata

- `background_mapping_policy`
- `pdb_mapping_policy`

### Required Business Parameters

- `background_user_count`
- `background_packet_kb`
- `background_period_ms`
- `pdb_user_count`
- `pdb_ms`
- `pdb_packet_kb`

This ensures that any result table can answer both:

- what `rho` point was intended
- what runnable traffic scene was actually used

## Rounding And Validity Rules

The mapper may need to round because simulator inputs are discrete or expected to be practical.

The implementation must therefore:

1. compute target offered load from target `rho`
2. solve for the mapped business parameter
3. round according to simulator-compatible rules
4. recompute `actual_rho_bg` and `actual_rho_pdb` from the rounded values

The report must never pretend rounded parameters still hit the target exactly.

If the rounded value would be invalid, the mapper must fail fast with a clear validation error.

## Config Shape

The config should move from a business-first shape to a rho-first shape.

Instead of:

- `background_packet_kb_values`
- `pdb_shapes`

the scan definition should use:

- `rho_bg_values`
- `rho_pdb_values`
- `pdb_ms_values`
- `mapping_policy`
- `capacity_reference`

The first-pass mapping-policy section should look conceptually like:

```json
{
  "mapping_policy": {
    "background": {
      "kind": "candidate_domain_solve_period",
      "background_user_count_values": [32, 40, 48],
      "background_packet_kb_values": [1.0, 1.5, 2.0],
      "background_period_ms_range": [4, 30],
      "anchor_background_user_count": 40,
      "anchor_background_packet_kb": 2.0
    },
    "pdb": {
      "kind": "candidate_domain_solve_packet",
      "pdb_user_count_values": [4, 8],
      "pdb_packet_kb_range": [0.5, 80.0],
      "anchor_pdb_user_count": 4
    }
  }
}
```

This makes the definition of the scan and the realization policy explicit and separable.

## Reporting Requirements

The report and CSV outputs should pivot around target and actual ratio coordinates.

They must no longer imply that:

- the scan was defined by packet sizes
- the chosen business parameters are the scan itself

At minimum, the ratio-based outputs must expose:

- target `rho`
- actual `rho`
- mapped business parameters
- policy comparison metrics

## Compatibility Strategy

The existing legacy-grid mode remains unchanged.

The current load-ratio mode should be replaced by the rho-first mode rather than keeping two incompatible flavors under the same conceptual name.

If migration support is needed during transition, it should be temporary and explicit.

## Success Criteria

This design is successful if:

1. the scan is defined directly in `rho` space
2. each `rho` point maps to concrete simulator inputs through an explicit policy
3. outputs expose both target and actual ratio coordinates
4. the first pass runs the intended `4 √ó 3 √ó 5 = 60` ratio-defined scene points
5. later work can add alternative mapping policies without redefining the meaning of the scan itself

## Out of Scope

This revision does not yet:

- sweep background user count
- sweep `PDB` user count
- sweep multiple background mapping policies
- sweep wireless parameters
- redesign the realization bank
