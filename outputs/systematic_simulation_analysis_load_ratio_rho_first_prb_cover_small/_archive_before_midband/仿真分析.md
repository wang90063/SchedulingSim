## 场景概览：
本文面向单小区上行仿真场景展开分析。小区内用户分为两类：一类是持续产生普通上行业务的背景用户，另一类是具有 PDB 时延约束的 PDB 用户。本文沿用 `outputs/systematic_simulation_analysis_load_ratio_rho_first_user_count_axes` 的报告结构，但本次结果是在修复背景业务发送周期映射后重新生成的，因此 PRB 利用率结论以本目录为准。

本轮仿真的目标是重新建立可信的 `rho` 扫描范围，使 baseline PRB 利用率覆盖低载、中载、重载和接近满载区域，同时保留重载下无法完全挽救的 PDB 场景。

## 仿真目的：
本轮实验有三个目的：

1. 验证修复后的背景业务 `background_period_ms` 是否真正进入仿真，从而使低 `rho_bg` 对应低 PRB 利用率；
2. 用较少场景覆盖约 `15%`、`30%`、`50%`、`70%`、`90%+` 和接近满载的 PRB 利用率区间；
3. 同时包含 easy、critical 和 heavy-overload 三类 PDB 负载，避免只看到可救回场景。

本轮继续采用“先归一化负载、再回映射业务参数”的方法。容量参考值保持为：

1. 背景侧容量参考值：`background_capacity_mbps = 66.03 Mbps`；
2. PDB 侧容量参考值：`pdb_capacity_mbps = 8.74 Mbps`。

业务注入负载比例覆盖范围如下：

| 类型 | 英文/代码变量 | 本轮覆盖范围 |
| --- | --- | --- |
| 背景用户负载占比 | `target_rho_bg` | `[0.03, 0.07, 0.12, 0.2]` |
| PDB 用户负载占比 | `target_rho_pdb` | `[0.03, 0.18, 0.55]` |

其中 `target_rho_pdb = 0.03` 对应 easy 区，`0.18` 对应临界压力观察区，`0.55` 专门用于重载和不可完全挽救场景。

#### 仿真结果
本轮实验以 `tail_append` 作为基线策略，以 `hopeless_tail_append` 作为改进策略。统计窗口为固定 `10 s`，PDB 满足率分母仅包含 `arrival_time < 10 s` 的 PDB 包。文中 `delta PDB satisfaction` 均为绝对增益，即 `proposed - baseline`，用百分点表达。

本轮共覆盖 `216` 个 scene points，对应 `1080` 条 paired realization rows 和 `2160` 次 policy runs。总体结果如下：

1. baseline PRB 利用率按 `target_rho_bg × target_rho_pdb × pdb_ms` 聚合后，均值覆盖 `14.2%` 到 `99.6%`，已经覆盖 30%、50%、70% 和接近满载区域。
2. 在 `216` 个场景中，改进策略正收益 `27` 个，持平 `168` 个，负收益 `21` 个；全场景平均 PDB 满足率增益为 `0.07` 个百分点。
3. `target_rho_pdb = 0.03` 的 `72` 个场景全部持平，baseline 平均满足率 `100.0%`，说明 easy 区没有可挽救空间。
4. `target_rho_pdb = 0.18` 的 baseline 平均满足率仍为 `100.0%`，本轮低 `rho_bg` 覆盖下它仍偏 easy，几乎没有收益。
5. `target_rho_pdb = 0.55` 形成重载区，baseline 平均满足率降至 `55.2%`，其中 `27/72` 个场景正收益、`21/72` 个场景负收益，平均增益约 `0.20` 个百分点，最高正收益约 `2.25` 个百分点，最低负收益约 `-1.08` 个百分点。这说明重载场景中既有少量可挽救边界包，也有相当多无法完全救回或不适合重排的点。

#### 核心结论
从工程使用角度看，本轮结果可以概括为一句话：`hopeless_tail_append` 的有效区间不是“所有高 PRB 利用率场景”，而是“PDB 业务已经把 baseline 推到 deadline 边界附近，同时仍存在可救包”的场景。PRB 利用率更像供给侧状态，用来判断小区资源是否紧张；`target_rho_pdb`、映射后的 PDB 业务模型，以及 baseline PDB 满足率，才决定 PDB 侧是否真的存在可救空间。

本轮修复后的 `rho_bg` 已经能够把 baseline PRB 利用率从约 `15%` 拉到接近满载，因此负载覆盖本身是有效的。但从收益分布看，算法收益并不会随 PRB 利用率单调增加。`target_rho_pdb=0.03` 和 `0.18` 在当前业务映射下基本都是 easy 点，baseline 已经能稳定满足 PDB；`target_rho_pdb=0.55` 才真正产生 deadline 压力，但其中又同时包含“可救边界包”和“重排后互相挤压”的情况，所以出现正收益、持平和负收益并存。

| PRB 负载状态 | PDB 负载/业务模型 | 观察到的收益 | 解释 |
| --- | --- | --- | --- |
| `15% ~ 40%`，资源明显宽松 | `target_rho_pdb = 0.03`，PDB 包约 `0.2 ~ 2.5 KB` | 无收益 | `72/72` 个场景全部持平，baseline PDB 满足率为 `100%`，没有需要救回的包 |
| `40% ~ 80%`，资源开始上升 | `target_rho_pdb = 0.18`，PDB 包约 `1.2 ~ 14.7 KB` | 仍基本无收益 | 本轮该负载在当前无线环境下仍偏轻，baseline 平均满足率仍接近 `100%` |
| `85% ~ 100%`，PDB 侧进入重载 | `target_rho_pdb = 0.55`，PDB 包约 `3.8 ~ 45.1 KB` | 有少量收益但不稳定 | baseline 平均满足率降至 `55.2%`，存在可救边界包，但也存在被重排挤出 deadline 的包 |
| 接近满载且 PDB 压力很高 | 高 `rho_bg` + `target_rho_pdb = 0.55` | 正收益、持平、负收益并存 | 总资源紧张时，重排本身会改变多个临界包的先后顺序，必须同时看 `rescued_count` 和 `harmed_count` |

因此，这轮结果支持三个判断。第一，修复后的 `rho_bg` 确实能够覆盖低载、中载、重载和接近满载区间，PRB 利用率口径已经可用。第二，低 PDB 负载和偏轻 PDB 业务模型没有收益，不是算法失效，而是 baseline 已经满足；此时继续提高 PRB 利用率也不一定产生收益。第三，当前 `hopeless_tail_append` 在 `target_rho_pdb=0.55` 的重载区能救回一部分边界包，但收益会被 harmed 包抵消，因此它更适合作为重载对照和机制分析对象；如果要寻找更稳定的正收益工作区，应在 `target_rho_pdb=0.30 ~ 0.45` 之间补扫，使 baseline PDB 满足率落入 `0.50 ~ 0.95` 的可救边界，而不是只看 `0.18` 和 `0.55` 两端。

从热力图可以得到以下结论：

- **结论一：轻载和低 PDB 负载区没有收益，原因是 baseline 已经完全满足。** 当 `target_rho_pdb=0.03` 时，`72/72` 个场景全部持平；即使部分点的 PRB 利用率已经不低，PDB 包仍然足够小、deadline 压力不足，baseline PDB 满足率保持 `100%`。这类业务模型不应被解释为算法没有提升能力，而应归类为无需干预的 easy 区。
- **结论二：本轮 `target_rho_pdb=0.18` 仍然偏轻，没有形成参考目录中的临界收益窗口。** 该负载下 PDB 包大小约 `1.2 ~ 14.7 KB`，在当前差点用户分布和 10s 周期发送口径下，baseline 平均满足率仍接近 `100%`。这说明 `rho_pdb` 不能脱离业务映射解释：同一个归一化负载如果映射成较小单包或较宽松周期，仍可能没有 deadline 边界。
- **结论三：`target_rho_pdb=0.55` 才形成真正的重载对照，但不是稳定收益区。** 该区域 baseline 平均满足率降至 `55.2%`，`27/72` 个场景正收益、`21/72` 个场景负收益、其余持平。最高正收益约 `2.25` 个百分点，最低负收益约 `-1.08` 个百分点，说明 proposed 确实改变了包级成败状态，但净效果取决于救回包和 harmed 包的差值。
- **结论四：业务模型轴仍然重要。** 在相同 `target_rho_pdb` 下，`pdb_user_count=8` 会把总 PDB 负载拆成更多小包，`pdb_user_count=4` 则形成更大的单包；`pdb_ms=50 / 100 / 300` 也不是单独改变周期，因为包大小会随 `rho_pdb` 反解同步变化。因此，热力图不应只按 PRB 利用率或 `pdb_ms` 做单调解释，而要同时看 `pdb_user_count`、`pdb_packet_kb` 和 baseline 满足率。

#### Packet-level case analysis
为了避免把 scene-level 满足率增益误读为“没有发生包级变化”，本文补充少量 packet-level transition 统计。这里不再使用旧的单个 target packet 排队/服务拆分口径，而是直接比较同一 realization、同一 PDB 包在 baseline 和 proposed 下的完成状态：

1. `rescued_count`：baseline 未按时完成，但 proposed 按时完成的 PDB 包数；
2. `harmed_count`：baseline 按时完成，但 proposed 未按时完成的 PDB 包数；
3. `net_rescued_count = rescued_count - harmed_count`；
4. `net_rescue_rate = net_rescued_count / total_pdb_arrivals`，它与 scene-level `delta PDB satisfaction` 等价。
5. `relative_gain_vs_baseline_satisfied = net_rescued_count / baseline_on_time`，表示相对于 baseline 已满足包规模的净提升；当 `baseline_on_time = 0` 时记为 `NaN`。

同时，本文对发生状态翻转的包补充二分时延拆解：

1. `waiting_delay_ms`：由于 `max_ue_per_slot` 候选窗口限制，UE 没有进入排序/调度候选集而形成的非服务等待；
2. `service_delay_ms = completion_delay_ms - waiting_delay_ms`：UE 进入候选排序后，到该包完成之间的服务过程时延；如果 grant 不足导致多轮调度，后续耗时也归入该服务过程；
3. `tx_airtime_ms = service_slot_count * slot_duration_ms`：包真正获得 grant 并发送 bits 的 U slot 时间，仅作为 `service_delay_ms` 内部的辅助解释；
4. `completion_delay_ms = waiting_delay_ms + service_delay_ms`：端到端完成时延，也是判断是否满足 PDB 的最终口径。

这里的 `harmed_count` 不是指 proposed 主动丢弃包，也不是不同随机场景之间的聚合误差，而是同一 realization、同一 `packet_id` 下的反事实状态翻转。它产生的根本原因是：`hopeless_tail_append` 虽然不会把已经判定完全不可救的包插到最前面，但对“插到某个靠前位置可满足”的包仍会做 constrained insert。这个判断只校验当前被重排包自身的完成时限，并不重新证明所有被后移的包仍然满足 PDB。因此，在 deadline 边界附近，提前一个包会占用有限的上行 grant，使另一个本来刚好能在 PDB 内完成的包被推迟到 PDB 之后。

为排除统计错配，本文对负收益代表 case 进行了逐包复核。该 case 的 5 个 seed 中，`seed=8/9/10` 分别出现 `rescued/harmed = 31/55`、`24/32`、`28/39`，`seed=7/11` 没有状态翻转，总计正好为 `rescued=83`、`harmed=126`。典型 harmed 包在 baseline 下的完成时延为 `98 ~ 100 ms`，在 proposed 下变为 `101 ~ 115 ms`，即不是大幅恶化，而是被资源重排推过了 `100 ms` PDB 阈值。这说明 harmed 主要发生在临界包上。

这个表解释了为什么当前 `hopeless_tail_append` 的 scene-level 增益较小。以最高正收益点为例，proposed 确实救回了 `143` 个 baseline 未满足包，但同时也使 `53` 个 baseline 原本满足的包转为未满足，净救回 `90` 个包；由于 10s 窗口内共有 `4000` 个 PDB 包，最终体现为 `2.25` 个百分点的满足率提升。如果换成相对 baseline 已满足包数的口径，则为 `90/746 = 12.06%`。`50 ms` 正收益点也是类似逻辑：救回 `332` 个包，但 harmed `200` 个包，净增只有 `132/8000 = 1.65` 个百分点；相对 baseline 已满足包数则为 `132/842 = 15.68%`。

时延拆解进一步说明，主口径应按“等待进入候选排序”和“进入候选排序后的服务过程”二分。最高正收益点中，rescued 包的等待时延从 `53.66 ms` 降到 `36.45 ms`，服务过程时延从 `173.87 ms` 降到 `41.09 ms`，其中真正发 bits 的 `tx_airtime` 只有 `4.15 ms` 和 `3.58 ms`，最终完成时延从 `227.53 ms` 降至 `77.54 ms`。但同一 case 的 harmed 包也存在：其等待时延从 `35.32 ms` 增至 `46.72 ms`，服务过程时延从 `56.15 ms` 增至 `62.77 ms`，完成时延从 `91.47 ms` 增至 `109.49 ms`，刚好被推过 `100 ms` PDB 阈值。

负收益点说明该策略不是单调改善策略。当 `target_rho_bg=0.07`、`target_rho_pdb=0.55`、`pdb_ms=100 ms` 时，proposed 救回 `83` 个包，但 harmed `126` 个包，净变化为 `-43` 个包，对应 `-1.08` 个百分点。从时延拆解看，rescued 包的完成时延从 `103.63 ms` 降至 `91.93 ms`，确实被救回；但 harmed 包的等待时延从 `4.71 ms` 增至 `6.27 ms`，服务过程时延从 `86.92 ms` 增至 `97.40 ms`，最终完成时延从 `91.63 ms` 增至 `103.67 ms`。这说明 queue reordering 会改变 deadline 边界附近包的先后顺序：如果被提前的包没有转化成足够多的按时完成，或者挤掉了更多原本可按时完成的临界包，scene-level 收益就会很小甚至为负。

因此，本轮结果应理解为：`hopeless_tail_append` 确实在 packet-level 上改变了部分 PDB 包的成败状态，但它的净收益取决于 `rescued_count` 和 `harmed_count` 的差值，而不是单看被救回包数。后续若要把该机制作为算法收益证据，报告中应同时给出 `rescued_count / harmed_count / net_rescued_count / net_rescue_rate`，并同步给出 `waiting_delay_ms / service_delay_ms / tx_airtime_ms / completion_delay_ms` 的拆解，避免只用单个完成时延或只用 raw rescued count 得出偏差结论。

#### 后续建议
本轮已经完成 PRB 利用率校准：修复后 `rho_bg` 可以有效把 PRB 利用率从约 15% 拉到接近 100%。为了找到更稳定的正收益区，下一轮不应继续盲目扩大 `rho_bg`，而应在本轮两端之间补扫 PDB 临界带：

1. 保留 `target_rho_bg = 0.12 / 0.20` 作为中高资源竞争背景；
2. 在 `target_rho_pdb = 0.30 / 0.40 / 0.45` 之间补点，寻找 baseline 满足率落入 `0.50 ~ 0.95` 的区间；
3. 继续保留 `target_rho_pdb = 0.55` 作为重载不可救对照；
4. 每个补点仍保留显式 `background_user_count × pdb_user_count` 轴，避免业务模型被 anchor 折叠。
