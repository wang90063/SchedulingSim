## 场景概览：
本文面向单小区上行仿真场景展开分析。小区内用户分为两类：一类是持续产生普通上行业务的背景用户，另一类是具有 PDB 时延约束的 PDB 用户。本文沿用 `outputs/systematic_simulation_analysis_load_ratio_rho_first_user_count_axes` 的报告结构，但本次结果是在修复背景业务发送周期映射后重新生成的，因此 PRB 利用率结论以本目录为准。

本轮仿真的目标是重新建立可信的 `rho` 扫描范围，使 baseline PRB 利用率覆盖低载、中载、重载和接近满载区域，同时保留重载下无法完全挽救的 PDB 场景。

## 仿真目的：
本轮实验有三个目的：

1. 验证修复后的背景业务 `background_period_ms` 是否真正进入仿真，从而使低 `rho_bg` 对应低 PRB 利用率；
2. 用较少场景覆盖约 `15%`、`30%`、`50%`、`70%`、`90%+` 和接近满载的 PRB 利用率区间；
3. 同时包含 easy、critical 和 heavy-overload 三类 PDB 负载，避免只看到可救回场景。

本轮继续采用“先归一化负载、再回映射业务参数”的方法。容量参考值保持为：

1. 背景侧容量参考值：`background_capacity_mbps = 66.03 Mbps`；
2. PDB 侧容量参考值：`pdb_capacity_mbps = 8.74 Mbps`。

业务注入负载比例覆盖范围如下：

| 类型 | 英文/代码变量 | 本轮覆盖范围 |
| --- | --- | --- |
| 背景用户负载占比 | `target_rho_bg` | `[0.03, 0.07, 0.12, 0.2]` |
| PDB 用户负载占比 | `target_rho_pdb` | `[0.03, 0.18, 0.30, 0.40, 0.45, 0.55]` |

其中 `target_rho_pdb = 0.03` 对应 easy 区，`0.18` 在本无线环境下仍偏轻；`0.30 / 0.40 / 0.45` 是本次补扫的临界带，用于填补 easy 和 heavy-overload 之间的空档；`0.55` 保留为重载和不可完全挽救对照。

#### 仿真结果
本轮实验以 `tail_append` 作为基线策略，以 `hopeless_tail_append` 作为改进策略。统计窗口为固定 `10 s`，PDB 满足率分母仅包含 `arrival_time < 10 s` 的 PDB 包。文中 `delta PDB satisfaction` 均为绝对增益，即 `proposed - baseline`，用百分点表达。

本轮在原有 `0.03 / 0.18 / 0.55` 三档基础上，补扫 `0.30 / 0.40 / 0.45` 三个中间 PDB 负载档位。合并后共覆盖 `432` 个 scene points，对应 `2160` 条 paired realization rows 和 `4320` 次 policy runs。总体结果如下：

1. baseline PRB 利用率按 `target_rho_bg × target_rho_pdb × pdb_ms` 聚合后，均值覆盖 `14.2%` 到 `99.6%`，仍然覆盖低载、中载、重载和接近满载区域。
2. 在 `432` 个场景中，改进策略正收益 `65` 个，持平 `307` 个，负收益 `60` 个；全场景平均 PDB 满足率增益为 `0.041` 个百分点。
3. `target_rho_pdb = 0.03` 和 `0.18` 基本全部属于 easy 区：baseline 平均满足率均约为 `100.0%`，没有可挽救空间。
4. `target_rho_pdb = 0.30` 只在高背景负载下开始接近 deadline 边界，baseline 平均满足率仍有 `97.4%`，整体收益接近 0，说明该档位大部分仍偏轻。
5. `target_rho_pdb = 0.40 / 0.45` 才形成更有分析价值的临界带。两档 baseline 平均满足率分别为 `87.6%` 和 `78.3%`，正收益、持平和负收益并存，最高正收益分别达到 `1.60` 和 `1.28` 个百分点。
6. `target_rho_pdb = 0.55` 仍是 heavy-overload 对照，baseline 平均满足率降至 `55.2%`，最高正收益 `2.25` 个百分点，但也存在 `-1.08` 个百分点的负收益点。这说明重载下确实能救回一部分包，但大量包已经不可救，且 harmed 包会抵消收益。

#### 核心结论
从工程使用角度看，本轮结果可以概括为一句话：`hopeless_tail_append` 的有效区间不是“所有高 PRB 利用率场景”，而是“PDB 业务已经把 baseline 推到 deadline 边界附近，同时仍存在可救包”的场景。PRB 利用率更像供给侧状态，用来判断小区资源是否紧张；`target_rho_pdb`、映射后的 PDB 业务模型，以及 baseline PDB 满足率，才决定 PDB 侧是否真的存在可救空间。

本轮修复后的 `rho_bg` 已经能够把 baseline PRB 利用率从约 `15%` 拉到接近满载，因此负载覆盖本身是有效的。但从收益分布看，算法收益并不会随 PRB 利用率单调增加。`target_rho_pdb=0.03` 和 `0.18` 在当前业务映射下基本都是 easy 点，baseline 已经能稳定满足 PDB；新增的 `target_rho_pdb=0.30 / 0.40 / 0.45` 显示，真正的临界带主要从 `0.40` 开始出现；`target_rho_pdb=0.55` 则进入 heavy-overload，对照价值强，但不宜单独代表算法的稳定收益区。

| PRB 负载状态 | PDB 负载/业务模型 | 观察到的收益 | 解释 |
| --- | --- | --- | --- |
| `15% ~ 40%`，资源明显宽松 | `target_rho_pdb = 0.03`，PDB 包约 `0.2 ~ 2.5 KB` | 无收益 | `72/72` 个场景全部持平，baseline PDB 满足率为 `100%`，没有需要救回的包 |
| `40% ~ 80%`，资源开始上升 | `target_rho_pdb = 0.18`，PDB 包约 `1.2 ~ 14.7 KB` | 仍基本无收益 | 本轮该负载在当前无线环境下仍偏轻，baseline 平均满足率仍接近 `100%` |
| `70% ~ 100%`，PDB 侧进入临界带 | `target_rho_pdb = 0.40 / 0.45`，PDB 包约 `2.7 ~ 36.9 KB` | 有收益但不稳定 | baseline 平均满足率降至 `87.6% / 78.3%`，开始出现可救边界包，同时也出现 harmed 包 |
| `85% ~ 100%`，PDB 侧进入重载 | `target_rho_pdb = 0.55`，PDB 包约 `3.8 ~ 45.1 KB` | 收益更大但更混杂 | baseline 平均满足率降至 `55.2%`，存在可救边界包，但也有大量两边都不满足的包 |
| 接近满载且 PDB 压力很高 | 高 `rho_bg` + `target_rho_pdb = 0.55` | 正收益、持平、负收益并存 | 总资源紧张时，重排本身会改变多个临界包的先后顺序，必须同时看 `rescued_count` 和 `harmed_count` |

因此，这轮结果支持四个判断。第一，修复后的 `rho_bg` 确实能够覆盖低载、中载、重载和接近满载区间，PRB 利用率口径已经可用。第二，低 PDB 负载和偏轻 PDB 业务模型没有收益，不是算法失效，而是 baseline 已经满足；此时继续提高 PRB 利用率也不一定产生收益。第三，`target_rho_pdb=0.30` 仍偏轻，真正值得重点分析的是 `target_rho_pdb=0.40 / 0.45`，尤其是 baseline 满足率落在约 `50% ~ 95%` 的场景。第四，`target_rho_pdb=0.55` 能给出最高正收益，但其中 heavy-overload 成分明显，报告中应把它作为“重载可救/不可救混合区”，而不是简单当作最佳工作区。

从热力图可以得到以下结论：

- **结论一：轻载和低 PDB 负载区没有收益，原因是 baseline 已经完全满足。** 当 `target_rho_pdb=0.03` 时，`72/72` 个场景全部持平；即使部分点的 PRB 利用率已经不低，PDB 包仍然足够小、deadline 压力不足，baseline PDB 满足率保持 `100%`。这类业务模型不应被解释为算法没有提升能力，而应归类为无需干预的 easy 区。
- **结论二：本轮 `target_rho_pdb=0.18` 仍然偏轻，没有形成参考目录中的临界收益窗口。** 该负载下 PDB 包大小约 `1.2 ~ 14.7 KB`，在当前差点用户分布和 10s 周期发送口径下，baseline 平均满足率仍接近 `100%`。这说明 `rho_pdb` 不能脱离业务映射解释：同一个归一化负载如果映射成较小单包或较宽松周期，仍可能没有 deadline 边界。
- **结论三：补扫后的 `target_rho_pdb=0.30 / 0.40 / 0.45` 填上了关键临界带。** `0.30` 的 baseline 平均满足率仍有 `97.4%`，整体偏 easy；`0.40` 和 `0.45` 的 baseline 平均满足率分别降至 `87.6%` 和 `78.3%`，开始稳定出现状态翻转包。该区域最高正收益分别为 `1.60` 和 `1.28` 个百分点，说明它比 `0.18` 更适合解释“什么时候有收益”。
- **结论四：`target_rho_pdb=0.55` 是重载对照，不是唯一的收益证据。** 该区域 baseline 平均满足率降至 `55.2%`，`27/72` 个场景正收益、`21/72` 个场景负收益、其余持平。最高正收益约 `2.25` 个百分点，最低负收益约 `-1.08` 个百分点，说明 proposed 确实改变了包级成败状态，但净效果取决于救回包和 harmed 包的差值；同时，大量两边都不满足的包会稀释 scene-level 增益。
- **结论五：业务模型轴仍然重要。** 在相同 `target_rho_pdb` 下，`pdb_user_count=8` 会把总 PDB 负载拆成更多小包，`pdb_user_count=4` 则形成更大的单包；`pdb_ms=50 / 100 / 300` 也不是单独改变周期，因为包大小会随 `rho_pdb` 反解同步变化。因此，热力图不应只按 PRB 利用率或 `pdb_ms` 做单调解释，而要同时看 `pdb_user_count`、`pdb_packet_kb` 和 baseline 满足率。

#### Packet-level case analysis
为了避免把 scene-level 满足率增益误读为“没有发生包级变化”，本文补充少量 packet-level transition 统计。这里不再使用旧的单个 target packet 排队/服务拆分口径，而是直接比较同一 realization、同一 PDB 包在 baseline 和 proposed 下的完成状态：

1. `stable_satisfied_count`：baseline 和 proposed 都按时完成的 PDB 包数；
2. `rescued_count`：baseline 未按时完成，但 proposed 按时完成的 PDB 包数；
3. `harmed_count`：baseline 按时完成，但 proposed 未按时完成的 PDB 包数；
4. `stable_missed_count`：baseline 和 proposed 都未按时完成的 PDB 包数；
5. `net_rescued_count = rescued_count - harmed_count`；
6. `net_rescue_rate = net_rescued_count / total_pdb_arrivals`，它与 scene-level `delta PDB satisfaction` 等价；
7. `relative_gain_vs_baseline_satisfied = net_rescued_count / baseline_on_time`，表示相对于 baseline 已满足包规模的净提升；当 `baseline_on_time = 0` 时记为 `NaN`。

四类包满足闭合关系：

`total_pdb_arrivals = stable_satisfied_count + rescued_count + harmed_count + stable_missed_count`

因此，`rescued_count` 和 `harmed_count` 只描述“跨过 PDB 阈值的状态翻转包”，并不代表其它包没有变化；其它包只是没有改变最终是否满足 PDB 的状态。

同时，本文对发生状态翻转的包补充二分时延拆解：

1. `waiting_delay_ms`：由于 `max_ue_per_slot` 候选窗口限制，UE 没有进入排序/调度候选集而形成的非服务等待；
2. `service_delay_ms = completion_delay_ms - waiting_delay_ms`：UE 进入候选排序后，到该包完成之间的服务过程时延；如果 grant 不足导致多轮调度，后续耗时也归入该服务过程；
3. `tx_airtime_ms = service_slot_count * slot_duration_ms`：包真正获得 grant 并发送 bits 的 U slot 时间，仅作为 `service_delay_ms` 内部的辅助解释；
4. `completion_delay_ms = waiting_delay_ms + service_delay_ms`：端到端完成时延，也是判断是否满足 PDB 的最终口径。

这里的 `harmed_count` 不是指 proposed 主动丢弃包，也不是不同随机场景之间的聚合误差，而是同一 realization、同一 `packet_id` 下的反事实状态翻转。它产生的根本原因是：`hopeless_tail_append` 虽然不会把已经判定完全不可救的包插到最前面，但对“插到某个靠前位置可满足”的包仍会做 constrained insert。这个判断只校验当前被重排包自身的完成时限，并不重新证明所有被后移的包仍然满足 PDB。因此，在 deadline 边界附近，提前一个包会占用有限的上行 grant，使另一个本来刚好能在 PDB 内完成的包被推迟到 PDB 之后。

为排除统计错配，本文对代表 case 采用逐包复核：每个 case 均在相同 seed、相同 realization bank、相同 PDB 包 `packet_id` 下比较 baseline 与 proposed 的完成状态。由于 PDB 包按 `arrival_time < 10 s` 进入统计窗口，部分 `pdb_ms = 300 ms` 场景的总包数不是整数千，但四类包闭合关系仍然成立。

这张表直接回答“除了 rescued 和 harmed 以外，其它包怎样了”。在 `target_rho_pdb=0.55` 的 best positive case 中，`3111/4000 = 77.8%` 的包两边都不满足，说明该点虽然有最高净收益，但整体已经偏 heavy-overload。相反，补扫后的 positive mid-band case 中，`6010/8000 = 75.1%` 两边都满足、`1530/8000 = 19.1%` 两边都不满足，同时仍有 `294` 个 rescued 和 `166` 个 harmed；这更接近“baseline 已经被推到 deadline 边界附近，但仍有可救空间”的临界场景。

这个表解释了为什么 scene-level 增益通常不大。以 positive mid-band case 为例，proposed 救回 `294` 个 baseline 未满足包，但同时造成 `166` 个 baseline 原本满足包转为未满足，净救回 `128` 个包；由于 10s 窗口内共有 `8000` 个 PDB 包，最终只体现为 `1.60` 个百分点的满足率提升。该点的 baseline 满足率为 `77.20%`，四类包分布相对均衡，因此比 heavy-overload 点更适合作为“临界带收益”代表。

时延拆解进一步说明，主口径应按“等待进入候选排序”和“进入候选排序后的服务过程”二分。positive mid-band case 中，rescued 包的等待时延从 `4.99 ms` 降到 `3.26 ms`，服务过程时延从 `49.51 ms` 降到 `40.27 ms`，完成时延从 `54.51 ms` 降至 `43.53 ms`。但同一 case 的 harmed 包也存在：等待时延从 `3.69 ms` 增至 `5.78 ms`，服务过程时延从 `41.50 ms` 增至 `48.26 ms`，完成时延从 `45.19 ms` 增至 `54.04 ms`，刚好越过 `50 ms` PDB 阈值。

negative mid-band case 说明该策略不是单调改善策略。当 `target_rho_bg=0.12`、`target_rho_pdb=0.45`、`pdb_ms=100 ms` 时，proposed 救回 `105` 个包，但 harmed `146` 个包，净变化为 `-41` 个包，对应 `-1.03` 个百分点。从时延拆解看，rescued 包的完成时延从 `103.99 ms` 降至 `96.05 ms`，确实被救回；但 harmed 包的完成时延从 `95.55 ms` 增至 `103.79 ms`，被推过 `100 ms` PDB 阈值。其机制仍然是 queue reordering 改变 deadline 边界附近包的先后顺序：如果被提前的包没有转化成足够多的按时完成，或者挤掉了更多原本可按时完成的临界包，scene-level 收益就会很小甚至为负。

因此，本轮结果应理解为：`hopeless_tail_append` 确实在 packet-level 上改变了部分 PDB 包的成败状态，但它的净收益取决于 `rescued_count` 和 `harmed_count` 的差值，而不是单看被救回包数。后续若要把该机制作为算法收益证据，报告中应同时给出 `rescued_count / harmed_count / net_rescued_count / net_rescue_rate`，并同步给出 `waiting_delay_ms / service_delay_ms / tx_airtime_ms / completion_delay_ms` 的拆解，避免只用单个完成时延或只用 raw rescued count 得出偏差结论。

#### 后续建议
本轮已经完成 PRB 利用率校准，并补上 `target_rho_pdb=0.30 / 0.40 / 0.45` 的临界带。后续不建议继续盲目扩大 rho 网格，而应围绕“稳定正收益”和“减少 harmed 包”做更聚焦的策略分析：

1. 把 `target_rho_pdb = 0.40 / 0.45` 作为主要收益分析带，优先关注 baseline 满足率约 `50% ~ 95%` 的场景；
2. 保留 `target_rho_pdb = 0.55` 作为 heavy-overload 对照，但报告中应明确其大量 `stable_missed_count` 会稀释增益；
3. 在策略层面增加 harmed guard，例如插入前估计被后移候选包的 deadline slack，避免为了救一个包而推迟更多临界包；
4. 后续若继续扩展业务模型，应优先改变 PDB 包形态和背景业务 burstiness，而不是继续加密已验证为 easy 的低 `rho_pdb` 区间。
