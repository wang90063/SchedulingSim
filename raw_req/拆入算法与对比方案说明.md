# 拆入算法与对比方案说明

## 1. 背景与问题定义

在 `5G MAC` 上行调度场景中，基站侧通常受限于单 `TTI` 内可处理的候选用户数上限，无法对全量活跃用户做精细排序。为控制实时计算开销，系统往往只对活跃队列头部前 `K` 个用户做排序与资源规划，其余用户继续留在队列中等待后续决策周期。

这类“队头候选窗口 + 轮转回插”机制在轻载场景下可以工作，但在高负载、尤其是边缘大包场景下会暴露出明显问题：

- 边缘用户由于频谱效率低，往往无法在一个调度周期内传完整个大包。
- 若该用户本轮发送后被直接放回队尾，则下一次重新进入候选窗口前需要等待多个控制决策周期。
- 在 `DSUUU` 帧结构下，`D/S` 时隙负责调度决策，`U` 时隙只执行既定发送，因此“重新进入候选集的等待时间”会被进一步放大。
- 当 `HOL` 持续增长而后续服务机会不足时，时延敏感业务容易突破 `PDB`。

因此，问题的核心并不是“是否允许插队”，而是：

> 在不打破现有候选用户规模约束的前提下，如何只对确有时延风险的用户实施最小必要前插，从而降低 `PDB` 违规率，同时避免无条件抢占带来的系统扰动。

## 2. 系统约束与符号说明

### 2.1 调度时序

本文假设上行采用 `DSUUU` 周期结构，一个周期长度记为 `T_cycle = 5 ms`：

- `D`：控制决策时隙，可生成后续一部分上行发送规划。
- `S`：控制决策时隙，可生成后续另一部分上行发送规划。
- `U1/U2/U3`：上行发送时隙，只执行既定 `PRB` 分配结果。

因此，相邻控制决策点之间的时间间隔并不均匀：

- 若当前位于 `D`，下一个控制决策点为 `S`，间隔 `1 ms`。
- 若当前位于 `S`，下一个控制决策点为下一周期的 `D`，间隔 `4 ms`。

### 2.2 基本符号

- `Q`：当前活跃用户队列。
- `|Q|`：当前队列长度。
- `K`：单次控制决策最多考虑的候选用户数。
- `u`：某个在本轮已被处理、但仍有剩余数据的用户。
- `HOL_u`：用户 `u` 当前队头包的 `Head-of-Line` 时延。
- `PDB_u`：用户 `u` 当前队头包的时延预算。
- `R_inst,u`：根据当前无线条件估计得到的瞬时可达速率。
- `R_avg,u`：用户 `u` 的历史平均吞吐量。
- `B_rem,u`：用户 `u` 当前队头包的剩余比特数。
- `B_srv,u`：本次控制决策已为用户 `u` 规划的发送比特数。
- `j`：用户重插入后在队列中的目标位置，`j = 0` 表示队首。

### 2.3 候选窗口规则

每次仅从队列头部取前 `K` 个用户进入候选集合，随后进行排序与资源规划。对一个未能一次发完的用户而言，其重插入位置将直接决定：

1. 它需要等待多少个控制决策点才能再次进入候选窗口。
2. 它在后续多少个 `DSUUU` 周期内才能完成剩余发送。
3. 它最终是否会突破 `PDB`。

## 3. 对比方案一：队尾回插

### 3.1 机制定义

队尾回插是最常见的基线策略。若用户在本轮被服务后仍有剩余数据，则不做任何时延风险判断，直接将其放回队尾。

该策略的优点是简单、稳定、易实现；缺点是当队列较长时，用户会经历确定性的多轮等待。

### 3.2 伪代码

```text
Algorithm TailAppend(u, Q)

Input:
  u : 当前轮已被处理但仍有剩余数据的用户
  Q : 当前活跃队列

1: if u.head_packet == NULL then
2:     append u to tail(Q)
3:     return
4: end if

5: if u.head_packet.remaining_bits <= 0 then
6:     return
7: end if

8: append u to tail(Q)
9: return
```

### 3.3 特点

- 优点：实现复杂度最低，不改变现有轮转顺序。
- 缺点：在高负载场景下，长包用户会因重复回到队尾而产生较大 `inter-service gap`。
- 适用场景：轻载场景、无严格时延要求的业务。

## 4. 对比方案二：激烈前插

### 4.1 机制定义

激烈前插是另一个极端对比方案。其规则是：只要用户在本轮服务后仍有剩余数据，则无条件将其插回队首，不再判断队尾是否安全，也不判断最晚安全位置。

这个方案能显著缩短该用户重新进入候选窗口的等待时间，但会导致队列顺序被持续打乱，容易产生中心用户饥饿、系统振荡和公平性下降。

### 4.2 伪代码

```text
Algorithm AggressiveFrontInsert(u, Q)

Input:
  u : 当前轮已被处理但仍有剩余数据的用户
  Q : 当前活跃队列

1: if u.head_packet == NULL then
2:     append u to tail(Q)
3:     return
4: end if

5: if u.head_packet.remaining_bits <= 0 then
6:     return
7: end if

8: insert u into Q before index 0
9: return
```

### 4.3 特点

- 优点：单用户时延压缩能力最强。
- 缺点：无条件抢占，系统扰动最大，容易破坏长期公平性。
- 适用场景：理论上界对比，不适合作为生产策略。

## 5. 我们的方法：受约束拆入

### 5.1 核心思想

我们的目标不是“尽可能往前插”，而是“只在必要时往前插，并且只前插到足够安全的位置”。

具体来说，受约束拆入遵循以下原则：

- 能尾插就尾插，保持与现有机制的兼容性。
- 只有当“按队尾回插会导致预测完成时间超过 `PDB`”时，才触发前插。
- 一旦需要前插，优先选择满足时延约束的最靠后位置，而不是无条件插到队首。
- 如果队列中不存在绝对安全位置，则至少将其推入下一候选窗口，使其尽快重新可见。

因此，该方法本质上是一种 **最小必要前插** 策略。

### 5.2 调度主流程

在每个控制决策点（`D` 或 `S`）上，整体调度框架如下：

```text
Algorithm SchedulingDecision(Q, K)

1: C <- first K users from queue Q
2: rank users in C by ePF weight
3: plan PRB allocation for ranked users
4: for each serviced user u in C do
5:     if u still has remaining bits then
6:         ConstrainedReinsert(u, Q)
7:     end if
8: end for
9: execute planned transmissions in subsequent U slots
```

其中，真正区分三种策略的唯一关键差异，在于第 `6` 步的回插方式。

## 6. `ePF` 动态权重公式

为决定候选窗口内哪些用户优先获得资源，我们采用增强型比例公平权重。对具有 `PDB` 约束的用户，权重定义为：

```math
W_u = \left(\frac{R_{inst,u}}{R_{avg,u}}\right)
\times
\left(\frac{HOL_u}{\max(1, PDB_u - HOL_u)}\right)
```

其中：

- `R_inst,u / R_avg,u` 体现当前无线条件与长期公平性的平衡；
- `HOL_u / (PDB_u - HOL_u)` 体现时延紧迫程度；
- 当 `HOL_u` 接近 `PDB_u` 时，第二项会快速增大；
- 当 `HOL_u >= PDB_u` 时，可直接赋予一个极大权重 `W_max`，强制其在候选集中优先。

若某类业务不参与 `PDB` 约束，则可退化为纯 `EPF` 形式：

```math
W_u = \frac{R_{inst,u}}{R_{avg,u}}
```

这保证了时延敏感业务与背景业务可以共享同一套排序框架，但拥有不同的紧迫性表达。

## 7. `safe index` 与安全性判定

### 7.1 基本思想

对某个本轮未发完的用户 `u`，我们并不直接问“该不该插队”，而是问：

> 若将 `u` 插回到队列中位置 `j`，它能否在 `PDB` 约束内完成剩余数据发送？

若答案为“能”，则位置 `j` 是安全的；若答案为“不能”，则位置 `j` 不安全。

在所有安全位置中，最靠后的位置记为 `safe_index(u)`。这就是“最晚仍可满足时延约束的位置”。

### 7.2 等待控制决策点的时间

若用户被插入到队列位置 `j`，则它距离重新进入候选窗口还需经历：

```math
h(j) = \left\lfloor \frac{j}{K} \right\rfloor
```

个控制决策跳数。

在 `DSUUU` 帧结构下，等待时间依赖于当前所处的控制相位：

```math
T_{wait}(D, h) =
\begin{cases}
0, & h = 0 \\
\sum_{n=1}^{h} g_D(n), & h > 0
\end{cases}
```

其中：

```math
g_D(n) =
\begin{cases}
1, & n \text{ 为奇数} \\
4, & n \text{ 为偶数}
\end{cases}
```

同理，若当前相位为 `S`：

```math
T_{wait}(S, h) =
\begin{cases}
0, & h = 0 \\
\sum_{n=1}^{h} g_S(n), & h > 0
\end{cases}
```

其中：

```math
g_S(n) =
\begin{cases}
4, & n \text{ 为奇数} \\
1, & n \text{ 为偶数}
\end{cases}
```

这两个表达式分别对应：

- `D -> S` 的等待为 `1 ms`
- `S -> next D` 的等待为 `4 ms`

### 7.3 剩余完成时间估计

在当前这轮调度中，用户 `u` 已经得到 `B_srv,u` 比特的规划服务量，因此本轮之后的剩余需求近似为：

```math
B'_{rem,u} = \max(B_{rem,u} - B_{srv,u}, 0)
```

若后续每次再次进入候选窗口时，仍能近似获得与本轮同等数量级的服务，则完成剩余数据大约还需要：

```math
N_{extra,u} =
\begin{cases}
\infty, & B_{srv,u} \le 0 \\
\left\lceil \frac{B'_{rem,u}}{B_{srv,u}} \right\rceil, & B_{srv,u} > 0
\end{cases}
```

个额外 `DSUUU` 周期。

因此，若将用户重插入到位置 `j`，其预测完成时间可写为：

```math
T_{comp}(u, j) = t_{now} + T_{wait}(\phi, h(j)) + N_{extra,u} \cdot T_{cycle}
```

其中：

- `t_now` 为当前控制决策时刻；
- `\phi \in {D, S}` 表示当前相位；
- `T_cycle = 5 ms`。

### 7.4 安全位置定义

设该包的理论截止时刻为：

```math
T_{ddl}(u) = t_{arrive}(u) + PDB_u
```

若考虑保护裕量 `T_guard`，则安全判定条件为：

```math
T_{comp}(u, j) \le T_{ddl}(u) - T_{guard}
```

因此，`safe_index` 可定义为：

```math
safe\_index(u) =
\max \left\{
j \mid 0 \le j \le |Q|-1,\;
T_{comp}(u, j) \le T_{ddl}(u) - T_{guard}
\right\}
```

若该集合为空，则说明队列中不存在绝对安全位置，此时只能采取“至少推进到下一候选窗口”的保底前插策略。

## 8. 受约束拆入伪代码

```text
Algorithm ConstrainedReinsert(u, Q, B_srv,u, t_now, phase, K)

Input:
  u       : 本轮已被处理但仍有剩余数据的用户
  Q       : 当前活跃队列
  B_srv,u : 本轮为用户 u 规划的发送比特数
  t_now   : 当前控制决策时刻
  phase   : 当前相位，phase in {D, S}
  K       : 单次候选用户上限

1: if u.head_packet == NULL then
2:     append u to tail(Q)
3:     return
4: end if

5: if u.head_packet has no PDB constraint then
6:     append u to tail(Q)
7:     return
8: end if

9: tail_index <- max(|Q| - 1, 0)

10: if PositionSafe(u, tail_index, B_srv,u, t_now, phase, K) then
11:     append u to tail(Q)
12:     return
13: end if

14: for j from tail_index downto 0 do
15:     if PositionSafe(u, j, B_srv,u, t_now, phase, K) then
16:         insert u into Q before index j
17:         return
18:     end if
19: end for

20: insert_pos <- min(K - 1, tail_index)
21: insert u into Q before index insert_pos
22: return
```

其中：

- 第 `10` 至 `12` 行表示“能尾插就尾插”；
- 第 `14` 至 `19` 行表示“从后向前搜索最靠后的安全位置”；
- 第 `20` 至 `21` 行表示“若无安全位置，则至少推进到下一候选窗口”。

因此，该算法既避免了激烈前插的过度抢占，又能避免队尾回插的确定性超时。

## 9. `DSUUU` 时序下的工作机制解释

受约束拆入之所以在 `DSUUU` 场景中有效，根本原因在于它显式考虑了“重新变得可调度”所需的控制等待时间，而不仅仅考虑静态队列位置。

其机制可以概括为以下三点：

### 9.1 真正决定时延的不是队列长度，而是控制决策跳数

在 `DSUUU` 下，用户即使已经在队列中前移，只要还未进入下一次 `D/S` 候选窗口，就无法获得新的资源规划。因此，回插策略的关键不是“前移了多少个位置”，而是“减少了多少个控制决策跳数”。

### 9.2 边缘长包对 `inter-service gap` 极为敏感

边缘用户由于 `MCS` 低、`bits/PRB` 小，即使本轮被调度，也往往只能发送大包的一部分。若分片之间的等待间隔过大，则 `HOL` 会在多个 `D/S/U/U/U` 周期中持续增长，导致：

- 排队等待时间明显放大；
- 第二次、第三次服务机会来得过晚；
- 最终在尚未完成全部分片前突破 `PDB`。

### 9.3 最小必要前插兼顾时延与公平性

激烈前插虽然可以进一步压缩目标用户时延，但会频繁打乱所有用户的相对顺序；队尾回插则几乎不对紧迫业务提供保护。受约束拆入位于两者之间：

- 对背景业务，若队尾本身已安全，则其行为自然退化为尾插；
- 对紧迫业务，仅在尾插不安全时才触发前插；
- 对必须前插的业务，也只把它推进到“仍可满足 `PDB` 的最晚位置”，而不是固定插到队首。

这种设计使得算法更适合作为工程可落地方案，而不是仅停留在理论上界。

## 10. 三种方案的机制对比小结

| 策略 | 回插规则 | 时延收益 | 公平性影响 | 工程扰动 |
| --- | --- | --- | --- | --- |
| 队尾回插 | 无条件插回队尾 | 最弱 | 最小 | 最小 |
| 激烈前插 | 无条件插回队首 | 最强 | 最大 | 最大 |
| 受约束拆入 | 仅在尾插不安全时前插到最晚安全位置 | 较强 | 可控 | 可控 |

从机制上看：

- 队尾回插代表“完全保守”；
- 激烈前插代表“完全激进”；
- 受约束拆入代表“按时延风险触发的最小必要重排”。

因此，若论文或汇报的目标是说明“为什么我们的方案优于基线，同时又比激进抢占更合理”，则上述三者正好构成一条清晰的对比链条。

## 11. 与当前仿真实现的对应关系

为便于后续实验复现，本文中的三种口径与当前仿真平台可以对应如下：

- 队尾回插：`tail_append`
- 我们的方法：`constrained_insert` 或论文口径下的 `business_aware_constrained_insert`
- 激烈前插：当前仓库中未作为正式策略实现，可作为额外对比方案加入

其中，当前主口径方案 `business_aware_constrained_insert` 的含义并不是“对某一类用户固定前插”，而是：

- 对具有宽松 `PDB` 的背景业务，队尾通常就是安全位置，因此行为自然退化为尾插；
- 对时延敏感、尤其是边缘长包业务，当队尾不安全时才会触发受约束前插。

这与本文提出的“最小必要前插”思想保持一致。
