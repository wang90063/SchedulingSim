# 调度算法机制走读与问题分析

yuque带图版：https://www.yuque.com/wisdom-2rvt5/irn2sg/fez84dkz8zkq4u80?singleDoc# 《调度算法机制走读与问题分析》
## 概述
本文基于已有的走读草稿进行整理，目标是把零散的实现描述转成一份可复用的研究笔记。重点不在穷尽所有代码细节，而在于把当前调度机制的输入、核心数据结构、关键执行路径、主要问题和潜在改造方向讲清楚，便于研发同学继续补证据，也便于评审场景快速对齐共识。

从当前材料看，现有机制并不是“调度失效”，而是在工程复杂度受限的前提下采用了近似轮转策略。它能够维持可运行的吞吐路径，但在大用户规模和时延敏感业务场景下，会把尾时延与抖动问题放大出来。

## 核心概念
- [[半轮询调度]]：当前用户调度更接近“候选 UE 轮转”，而不是每个 TTI 都做全量精排。
- [[PDB时延预算]]：决定时延敏感流量能否承受当前排队位置的关键约束。

## 工作原理
### 1. 输入
调度输入本质上是“当前有待服务需求的 UE 集合”。在现有实现里，这些 UE 不是一次性全量排序，而是先选取候选集合，再逐步取出处理。

下行关键结构包括：
- `gp_dl_ue_non_zero_queue_load_and_harq_list`：保存新传且 queue load 大于 0 的 UE。
- `g_dl_qos_sch_priority`：与 HARQ 优先处理相关的队列。

上行关键结构包括：
- `gp_non_zero_bsr_list`：保存 BSR 非零的 UE。
- `g_ul_qos_sch_priority`：与上行重传相关的优先队列。

每个链表节点都能回到 `ue_context`，再通过 `lc_id` 找到各逻辑信道的调度参数，例如 `mcs index`、`5qi params` 等。这些信息会被用于计算 LC 权重。

### 2. 核心数据结构
本轮 TTI 的核心中间产物是 `priority_lc_tree`（每个下行时隙输出资源映射情况）。它是每个调度周期临时构建的优先级输出树，用来把“本轮哪些 LC 更重要”表达出来，供后续资源分配模块按顺序消费。

下行路径中，还存在两个分层优先树：
- `gp_dl_qos_high_priority_lc_tree`：主要承接 minBR/GBR 约束下优先级更高的 LC（`available token`>0）。
- `gp_dl_qod_drb_priority_lc_tree`：主要承接 mbr/ambr 等第二轮候选 LC（`delta token`  >0)。

关键计算入口为：
- 下行：`dl_qos_calc_lc_weight_and_insert`
- 上行：`ul_qos_calc_lc_weight_and_insert`
根据HOL，PDB，rb_size ，5QI weight等参数计算每个lc的重要性指标

### 3. 关键流程
可以把当前机制概括成“候选 UE 轮转 + LC 权重树输出”。

1. 初始化当前 TTI 的候选预算 `candidate_ue`（半轮询机制）。
2. 当 `candidate_ue > 0` 时，从全局链表中取出一个 UE 节点。
3. 检查 UE 是否处于合法激活状态，非法则跳过。
4. 遍历该 UE 下的各个 LC，检查是否有数据、token 是否满足等条件。
5. 满足条件的 LC 调用 `*_calc_lc_weight_and_insert` 计算权重，并写入 `priority_lc_tree` 。
6. `candidate_ue` 递减 1，如果该用户有lc进入priority_lc_tree，那么该节点的Node移到全局队列的尾部，跳至步骤2继续处理下一个 UE

### 4. 输出
输出并不是最终的 RB 分配结果，而是按 LC 重要性排好序的 `priority_lc_tree`。资源分配模块后续从这棵树中按顺序取出候选 LC，为其分配 RB，因此这棵树的组织方式会直接影响时延表现和公平性表现。


## 当前问题：半轮询机制的约束与影响
### 解释什么是半轮询机制
在现有实现里，在调度时，UE 不是全量排序，而是只对前K个用户排序维，即候选集合，如果一个用户的某个lc符合被调度条件，那么它就会在全局链表中，被放在最后一个

### 原因

当前采用半轮询（而非全量用户排序）是受限于**软件实现复杂度与硬件规格**的妥协之举。 如果小区用户数过多（当前上限可达 400 UE），对全量 UE 进行权重计算会导致单个 TTI 内调度超时（DSP 算力溢出）。因此，只能退而求其次，通过 `candidate_ue`（例如默认 16 个用户）进行阶段性截断轮转，以此控制 CPU 消耗和实时性。
### 核心痛点
半轮询机制面临的最大问题不是“偶尔排队”，而是**一旦用户被重新挂载到链表尾部，其新到达的包将面临确定性的多个调度周期等待**。

**场景推演：极度拥塞下的长尾时延测算**
-   **前置条件**：当前小区活跃用户约 `96` 人，单 TTI 调度上限 `candidate_ue = 16`，一个调度调度周期(约 5个 TTI) 耗时约为 `0.5`  `1 ms`。
- **轮转周期**：某 UE 本轮被调度后放至队尾，需要等待前面 `96 - 16 = 80` 个用户先完成调度。
-  **等待时延**：如果一个高优新包在该 UE 刚被移到队尾时到达，它将无法在当前周期发送,该包必须死等 `(96 / 16 - 1) = 5` 个调度周期才能重新拿回调度权。带来的额外空口排队时延约为 `5 × (0.5 ~ 1 ms) = 2.5 ~ 5 ms`，最终时延是 `12.5 和 25 ms`**。

对于 URLLC (超可靠低延迟通信) 或云游戏等极低 PDB 业务，这种机制引发的时延抖动是致命的。

## 解决措施
 ### 方案一：可调度用户数自适应调节

- **思路**：打破固定的 `candidate_ue` 配置。根据小区实时活跃用户数、用户 PDB 紧迫程度、以及当前的软硬件 CPU 算力水位，动态推导出最优的可调度用户数上限。
- **局限与风险**：治标不治本。受限于底层硬件处理时延的物理天花板，在默认 16 个的基础上能上浮的空间非常有限，无法彻底解决满载下的排队问题。

方案二：可调度用户数自适应调节
基于 PDB 的受约束插队：当前更可行的方向不是全量精排，而是在现有链表轮转机制上增加“受约束插队”能力。

思路如下：
		1. 为候选 UE 或关键 LC 估算最晚可接受服务轮次，依据包括 `PDB`、当前排队位置、总活跃用户数和 `candidate_ue` 上限。
		2. 不再对所有候选统一尾插，而是允许时延紧迫流量插入到“不会突破 PDB 的最晚安全位置”之前。
 
```
def insert_ue_with_pdb_constraint(global_ue_list, current_ue, candidate_ue_limit, cycle_time_ms):
		    """
		    基于 PDB 约束的插队逻辑 (简化版)
		    :param global_ue_list: 全局候选用户链表
		    :param current_ue: 当前刚处理完且仍有数据待传的 UE
		    :param candidate_ue_limit: 每轮 TTI 允许处理的最大 UE 数 (即 candidate_ue)
		    :param cycle_time_ms: 经历一轮调度的预估时间 (例如 1ms)
		    """
		    # 1. 获取该 UE 当前最紧迫的 PDB (取所有有数据的 LC 中最小的 PDB)
		    min_pdb_ms = get_min_pdb(current_ue)
		    
		    # 2. 倒推计算：以当前的 PDB 预算，最多还能等待几个调度周期？
		    max_wait_cycles = int(min_pdb_ms / cycle_time_ms)
		    
		    # 3. 换算安全位置索引：最晚必须在队列的哪个位置之前被处理？
		    # 例如：最多等 3 轮，每轮处理 16 个，则最晚必须排在第 48 个位置之前
		    safe_index = max_wait_cycles * candidate_ue_limit
		    
		    # 4. 执行受控插队判断
		    current_queue_size = len(global_ue_list)
		    
		    if current_queue_size <= safe_index:
		        # 场景 A：队列当前长度较短，即使放队尾也能在 PDB 超时前被轮到。
		        # 行为：直接尾插 (与原版逻辑一致，降低操作开销)
		        global_ue_list.append(current_ue)
		    else:
		        # 场景 B：队列较长，直接放队尾会导致 PDB 超时丢包！
		        # 行为：执行插队，强行将其插入到安全索引位置
		        global_ue_list.insert(safe_index, current_ue)
```
